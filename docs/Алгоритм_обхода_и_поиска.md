# Алгоритм обхода доменов и поиска на страницах

Документ описывает, как работает скринер (`scan.py` + пакет `scanner/`):
как он обходит сайты СПбПУ и как ищет на каждой странице упоминания
организаций/лиц из перечня. Методическая основа поиска — `search.md`
(нормализация → точный поиск Aho-Corasick → нечёткий поиск).

```
домены.xls (217 хостов)
        │
        ▼   [ЧАСТЬ 1: ОБХОД ДОМЕНОВ]
  перечисление страниц (sitemap.xml / BFS)  →  загрузка страницы
        │                                            │
        │                                            ▼   [ЧАСТЬ 2: ПОИСК]
        │                              нормализация → Aho-Corasick → fuzzy
        │                                            │
        ▼                                            ▼
   следующая страница                      находка → таблица scan_findings (БД)
```

## Часть 1. Обход доменов

Точка входа — `scan.py`, сетевые операции — `scanner/crawler.py`.

### 1.1. Список целевых доменов
- Источник истины — файл `домены.xls` (217 доменов `*.spbstu.ru`: высшие
  школы, институты, центры, кафедры, журналы, сервисы и т.д.).
- `load_hosts_file()` читает колонку «Домен», приводит к чистому хосту
  (без схемы/пути), нижний регистр, дедуп, фильтр по `domain_suffix=spbstu.ru`.
- Приоритет у файла; если его нет — авто-обнаружение по странице структуры
  (`discover_unit_hosts()` парсит ссылки `*.spbstu.ru` со страницы `/structure/`).

### 1.2. Перечисление страниц каждого хоста
Вместо «слепого» обхода по ссылкам используется **карта сайта** —
самый вежливый и полный способ узнать все адреса:
- `collect_sitemap_urls(host)` запрашивает `https://<host>/sitemap.xml`;
- `iter_sitemap_urls()` рекурсивно разворачивает **карты-индексы**
  (`<sitemapindex>` → вложенные карты), поддерживает **gzip** (`*.xml.gz`),
  извлекает адреса из тегов `<loc>`;
- если у хоста нет sitemap — включается **BFS-фолбэк** (`SiteCrawler`):
  обход по ссылкам в пределах хоста с ограничением глубины (`max_depth`).

### 1.3. Чередование хостов (round-robin)
Адреса всех хостов **перемешиваются по кругу** (`_round_robin`):
`[host1/стр1, host2/стр1, host3/стр1, host1/стр2, …]`. Благодаря этому
ограниченный прогон (лимит страниц) охватывает МНОГО подразделений, а не
исчерпывает один крупный сайт.

### 1.4. Вежливость и безопасность нагрузки
- **robots.txt** соблюдается отдельно для каждого хоста — кэш `RobotsCache`
  (один разбор на поддомен; `can_fetch` для нашего User-Agent);
- **задержка** между запросами (`delay_seconds`);
- **пропуск бинарных** ссылок (pdf, изображения, архивы, css/js);
- **лимиты**: `max_pages_total` (всего) и `max_pages_per_host` (на хост);
  `0` = без предела (обойти всё);
- единый HTTP-слой `loaders.fetch`: «браузерный» User-Agent, таймауты, ретраи,
  опция `verify_ssl` (сертификаты НУЦ РФ).

### 1.5. Возобновляемость
Каждый пройденный URL дописывается в `state_file` (`output/scan_state.txt`).
При повторном запуске уже просканированные адреса пропускаются (`resume`) —
длительный обход можно безопасно прерывать и продолжать.

**Псевдокод обхода:**
```
hosts = load_hosts_file("домены.xls")            # 217 доменов
urls  = round_robin([sitemap(h) for h in hosts]) # перечисление страниц
for url in urls:
    if url in done or not robots.allowed(url): continue
    page = fetch_page(url)                        # загрузка + извлечение текста
    scan_page(page)                               # ← Часть 2
    done.add(url); sleep(delay)
    if scanned >= max_pages_total: break
```

## Часть 2. Поиск на странице

Реализация — `scanner/matcher.py`. Подход двухэтапный (см. `search.md`).

### 2.1. Извлечение текста страницы
`extract_visible()` берёт не только видимый текст, но и **адреса ссылок (href)**
и атрибуты **alt/title** — чтобы ловить соцсети-иконки без подписи
(напр. ссылку на `facebook.com` в футере).

### 2.2. Нормализация (этап 1 search.md)
`normalize()` приводит и текст, и имена справочника к единому виду:
единый регистр, **ё → е**, удаление кавычек/дефисов/пунктуации, схлоп пробелов.
Пример: `ООО «Стэнфорд»` → `ООО СТЭНФОРД`.

### 2.3. Справочник организаций (этап 2 search.md)
- основной перечень читается **из базы данных** (таблица `registry_all`);
- **алиасы** (`matching.aliases`) — несколько написаний одной сущности на
  латинице/кириллице (Instagram / Инстаграм / `instagram.com` / Facebook …);
- **стоп-слова** отсекают одиночные общеупотребительные слова («объединение»);
- **фильтр** (`matching.drop`) убирает физлиц-террористов/экстремистов —
  по ним поиск даёт лишь ложные тёзки (в БД они остаются, в скрининг не идут).

### 2.4. Точный поиск — Aho-Corasick (этап 3 search.md)
`NameMatcher` строит автомат Ахо-Корасик по ВСЕМ именам справочника и находит
их вхождения **за один проход** по тексту страницы:
- сложность `O(длина_текста + число_совпадений)` — не зависит от размера
  словаря, поэтому масштабируется на десятки тысяч имён;
- **проверка границ слов**: совпадение должно быть отделено пробелом с обеих
  сторон, поэтому «ЯШИН» не находится внутри «КУЗЯШИН».

### 2.5. Нечёткий поиск — Fuzzy (этап 4 search.md)
`FuzzyMatcher` (RapidFuzz, Damerau–Levenshtein) ловит **опечатки и неизвестные
варианты**, которые Aho-Corasick (только точные совпадения) пропускает
(«стэнфорд» → «стэнфордд»). Из текста формируются окна по 1..N слов и
сравниваются со словарём; совпадения с оценкой ≥ порога фиксируются.
Включается опцией `matching.fuzzy.enabled` (для полного обхода выключено —
дорого; для точечных проверок — включено, см. `scan_demo.yaml`).

### 2.6. Достоверность и дедупликация
- `assess_confidence()` оценивает каждое совпадение: организации и редкие
  имена — **высокая**, частые ФИО («Иванов Александр Владимирович») — **низкая
  (возможен тёзка)**;
- одна сущность на странице фиксируется один раз (даже если совпало несколько
  её написаний, напр. «Facebook» и `facebook.com`).

### 2.7. Запись результата
Находки (`Finding`: сущность, категория, достоверность, URL, заголовок,
контекст) пишутся **только в базу данных** — таблица `scan_findings`
(`scanner/report.py → write_findings`). Таблица перезаписывается на каждом
флеше, поэтому всегда отражает текущий прогресс; строки отсортированы по
убыванию достоверности. CSV/XLSX в проекте не используются.

**Схема поиска на странице:**
```
текст страницы (+ href, alt)
        │ нормализация (регистр, ё→е, пунктуация)
        ▼
   Aho-Corasick (точно)
        │ найдено?
   ┌────┴─────────────┐
   да                 нет → Fuzzy (RapidFuzz) → похожие
   │                                              │
   ▼                                              ▼
 проверка границ слов · оценка достоверности · дедуп
        │
        ▼
   таблица scan_findings (PostgreSQL)
```

## Где что настраивается (`config.yaml`)
| Параметр | Значение |
|---|---|
| `scan.hosts_file` | файл целевых доменов (`домены.xls`) |
| `scan.max_pages_total` / `max_pages_per_host` | лимиты страниц (0 = всё) |
| `scan.delay_seconds`, `respect_robots`, `resume` | вежливость и возобновление |
| `matching.registry_db` | таблица перечня в БД (`registry_all`) |
| `matching.aliases`, `matching.drop`, `matching.fuzzy` | справочник, фильтры, fuzzy |
| `report.db_table` | таблица находок (`scan_findings`) |

Ключевые модули: `scan.py` (оркестрация), `scanner/crawler.py` (обход, sitemap,
robots, домены), `scanner/matcher.py` (нормализация, Aho-Corasick, fuzzy,
достоверность), `scanner/report.py` (запись находок в БД).
