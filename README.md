# Registry Merger — сбор и объединение госреестров РФ

Модульный Python-инструмент для автоматического сбора данных из официальных
государственных перечней (комплаенс / AML-проверка контрагентов), их очистки,
нормализации, объединения и выгрузки в Excel / CSV / SQLite.

## Настроенные источники

| # | Реестр | URL | Формат | Загрузчик | Записей |
|---|--------|-----|--------|-----------|--------|
| 1 | Минюст — экстремистские организации | `minjust.gov.ru/ru/documents/7822/` | нумерованный HTML-список | `html_list` | ~141 |
| 2 | ФСБ — террористические организации | `fsb.ru/fsb/npd/terror.htm` | HTML-таблица `<table>` | `web` | ~72 |
| 3 | Росфинмониторинг — террористы и экстремисты | `fedsfm.ru/documents/terrorists-catalog-portal-act` | HTML-список из `<li>` (организации **и** физлица) | `html_list` (`item_selector: li`) | ~22 700 |
| 4 | Минюст — иностранные агенты | `reestrs.minjust.gov.ru/rest/registry/{id}/values` | JSON API с пагинацией | `api` (POST, offset/limit) | ~1 200 |

Проверено на «живых» источниках: суммарно ~24 160 записей до дедупликации,
~24 080 после. Все три формата на выходе создаются корректно.

> **Важно по двум «хитрым» источникам:**
> - **Росфинмониторинг** публикует 4 раздела (Нацчасть/Международная ×
>   Организации/Физлица). Чтобы забрать *все* записи, включая физических лиц,
>   парсятся все элементы `<li>` (`item_selector: li`), а не один блок текста.
> - **Иностранные агенты** грузятся не файлом, а JSON-виджетом
>   `reestrs.minjust.gov.ru`. Загрузчик `api` постранично (POST `offset/limit`)
>   забирает все ~1200 записей; технические поля `field_N` расшифрованы в
>   `mapping` (взяты из `GET /rest/registry/{id}/info`).

## ⚠️ Особенности госсайтов (важно для запуска)

1. **TLS-сертификаты НУЦ РФ** отсутствуют в системном хранилище → во всех
   источниках стоит `verify_ssl: false`. Это снижает защиту соединения и
   допустимо **только** для перечисленных доверенных официальных доменов.
2. Без **браузерного User-Agent** часть сайтов таймаутит — он задан в
   [loaders/http.py](loaders/http.py) по умолчанию, плюс ретраи.
3. **Вёрстка меняется.** При сбое парсинга правьте *конфиг, а не код*:
   `container_selector`, `name_until`, `extract` (для `html_list`),
   `columns` / `header` / `table_index` (для `web`).
4. **Реестр иноагентов** грузится через JS (`href="#"`), статической ссылки
   на странице нет. В конфиге указан **прямой URL XLSX**, но его имя содержит
   меняющийся хеш. Когда ссылка вернёт 404:
   1. откройте страницу реестра иноагентов;
   2. нажмите «Загрузить реестр» и скопируйте URL скачанного файла;
   3. вставьте его в `path` источника №4 в `config.yaml`.
   Раздел иноагентов состоит из нескольких файлов (юрлица, незарегистрированные
   объединения, физлица, иностранные структуры) — каждый добавляется как
   отдельный источник по аналогии.

## Установка и запуск

Виртуальное окружение уже создано в `.venv` (можно пересоздать):

```bash
cd registry_merger
python3 -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python main.py --config config.yaml
```

Результаты — в `output/` (`registry_merged.xlsx|csv|db`), журналы — в `logs/`.

**Итоговая таблица — 3 колонки** (настраивается в `config.yaml → merge`):

| № п/п | Полное наименование (… ) / ФИО «Псевдоним» (…) | Тип (категория) |
|------|-----------------------------------------------|-----------------|

- `№ п/п` — сквозная нумерация (`merge.add_row_number`);
- наименование/ФИО — поле `full_name`, переименованное через `merge.column_titles`;
- `Тип (категория)` — поле `category` (Экстремистская организация /
  Террористическая организация / Иностранный агент / перечень РФМ).

**Сущность в нескольких реестрах.** Одно лицо/организация может числиться
сразу в нескольких перечнях (напр. человек — и в перечне РФМ, и как
иностранный агент). Дедупликация схлопывает такие записи в ОДНУ строку, но
колонка `category` через `merge.combine_columns` собирает ВСЕ категории:
`Террорист/экстремист (перечень РФМ); Иностранный агент`. Поэтому строк всего
~23 800, но «Иностранный агент» встречается в 968 из них (= все действующие
иностранные агенты, как в API Минюста).

Внутренний (промежуточный) состав полей шире — `full_name`, `birth_date`,
`inn`, `basis`, `address`, `reg_date`, `exclusion_date`, `subject_type`,
`source` — он используется для нормализации, фильтрации и дедупликации, а в
финальную таблицу выводятся только колонки из `merge.output_columns`
(`strict_columns: true`).

**Удаление исключённых.** Записи удаляются по двум правилам качества (оба
срабатывают ДО дедупликации, чтобы исключённая запись не вытеснила активную):
- `quality.drop_if_filled: [exclusion_date]` — если заполнена дата исключения
  из реестра (у иностранных агентов — `field_5_dt`). На последнем прогоне так
  удалён 251 иностранный агент (совпадает с данными API Минюста).
- `quality.drop_if_matches: {full_name: "исключен[ао]\\b"}` — строки-заглушки
  вида «Организация исключена в связи с ликвидацией …».

## Типы источников и их опции

- **`html_list`** — нумерованный список `N. текст`. Опции: `item_selector`
  (CSS-селектор элемента-записи, напр. `li` — надёжнее «нарезки» текста, когда
  список разбит на разделы), `container_selector` (CSS-контейнер),
  `basis_markers` (ключевые слова основания — `решение`, `приговор`…; имя
  отрезается перед `(<ключевое слово>`, что корректно работает с названиями,
  содержащими собственные скобки и кавычки), `name_until` (запасной
  regex-разделитель, напр. `,` для ФИО), `extract` (`{колонка: regex}` — напр.
  дата рождения, ИНН), `min_len`.

  > Почему `basis_markers`, а не «резать по первой `(`»: название
  > «Общественное объединение **(движение)** «Омская организация …»
  > **(решение** Омского облсуда …)» содержит свою скобку `(движение)`.
  > Разрыв по первой `(` обрезал бы имя до «Общественное объединение»;
  > разрыв по `(решение` сохраняет название целиком.
- **`api`** — JSON по HTTP. Опции: `method`, `headers`, `body`,
  `records_path` (путь до массива записей), а для постраничной выгрузки —
  `paginate` (`offset_field`, `limit_field`, `page_size`, `total_path`).
- **`web`** — HTML-таблица. Опции: `table_index`, `header`, `match`,
  `columns` (позиционное переименование), `skiprows`.
- **`excel`** — путь к файлу, прямой URL `.xlsx`, либо страница со ссылкой
  (`find_link_on_page: true`). Опции `sheet_name`, `header`, `engine`.
- **`csv` / `json` / `xml` / `api`** — стандартные табличные/иерархические
  источники (см. примеры в коде и комментарии в `config.yaml`).

Сетевые опции (`verify_ssl`, `user_agent`, `headers`, `timeout`, `retries`,
`retry_backoff`, `encoding`) поддерживаются всеми загрузчиками по сети.
`constants` (в `options`) добавляет статические колонки к каждой записи
источника — так размечаются `category` и `subject_type`.

## Скринер сайтов (поиск вхождений перечня)

Второй инструмент — `scan.py` — ищет на целевом сайте упоминания лиц/
организаций из собранного перечня (комплаенс-скрининг сайтов СПбПУ).

```bash
python scan.py --config scan_config.yaml
```

Как устроено (`scanner/`):
- **`matcher.py`** — нормализует имена (регистр, кавычки, дефисы) и строит
  автомат **Ахо-Корасик**: вхождения всех ~24 000 имён находятся за ОДИН
  проход по тексту страницы, а не 24 000 отдельными поисками. Проверка границ
  слов исключает совпадения внутри слов («Яшин» ≠ «Кузяшин»). Стоп-слова
  отсекают одиночные общеупотребительные слова («Объединение», «Сирена»),
  дающие шум. Оценка **достоверности**: организации и редкие имена — высокая,
  частые ФИО («Иванов Александр Владимирович») — низкая («возможен тёзка»).
  **Алиасы** (`matching.aliases`) — несколько написаний одной запрещённой
  сущности на латинице и кириллице: напр. Meta в перечне записана длинной
  фразой, а на сайтах пишут `Instagram` / `Инстаграм` / `Facebook` / `Фейсбук`
  или ставят ссылку `instagram.com`. Поиск ведётся в видимом тексте И в
  адресах ссылок/alt-атрибутах (чтобы ловить соц-иконки без подписи).
- **`discovery.py`** — обнаруживает все подразделения и их поддомены из
  официальной страницы структуры СПбПУ (институты, школы, лаборатории,
  центры, кафедры, сервисы).
- **`sitemap.py`** — перечисляет страницы каждого хоста через `sitemap.xml`
  (с рекурсией по картам-индексам и gzip) — вежливо и полно, без слепого BFS.
- **`crawler.py`** — загрузка страниц, извлечение текста, BFS-фолбэк для
  хостов без sitemap, кэш `robots.txt` по каждому хосту, паузы между запросами.
- **`report.py`** — отчёт в CSV/XLSX: сущность, категория, достоверность, URL,
  заголовок и контекст; XLSX с автошириной и подсветкой достоверности.

### Режимы обхода (`scan.mode`)
- **`single`** — один хост (BFS по ссылкам).
- **`structure`** — ВСЯ структура СПбПУ: обнаружение подразделений →
  перечисление страниц по sitemap всех ~52 хостов → обход.

> **Масштаб.** Структура СПбПУ — это **~73 000 страниц** на 36 поддоменах с
> sitemap (плюс 16 хостов под BFS-фолбэк). Полный обход (`max_pages_total: 0`)
> — многочасовой batch. Предусмотрены: лимиты (`max_pages_total` /
> `max_pages_per_host`), **round-robin** по хостам (ограниченный прогон
> охватывает много подразделений), и **возобновляемость** (`resume: true` —
> повторный запуск продолжает с места остановки, безопасно прерывать).
> Тестовый прогон (250 страниц, 34 поддомена) — 2 совпадения, оба
> однофамильцы-сотрудники, помечены «низкая (возможен тёзка)».

Параметры обхода (режим, структура, лимиты, задержки) и перечня — в
[scan_config.yaml](scan_config.yaml).

- Планировщик (macOS launchd): [launchd/com.spbpu.registry-scan-weekly.plist](launchd/com.spbpu.registry-scan-weekly.plist)
  — каждое воскресенье в 02:00, с пониженным приоритетом (Background/Nice).
  Установка:
  ```bash

  launchctl load ~/Library/LaunchAgents/com.spbpu.registry-scan-weekly.plist
  ```
- Альтернатива (cron): `0 2 * * 0 /Users/artemy/Desktop/практ/registry_merger/weekly_scan.sh`

> Полный обход (~73 000 страниц) длится несколько часов — поэтому недельная
> периодичность. Задержка между запросами (`delay_seconds`) и приоритет

> начальник отдела по противодействию коррупции «Иванов Александр Владимирович»,
> однофамилец лица из перечня РФМ; система пометила его как достоверность
> «низкая (возможен тёзка)». Точную идентификацию физлиц даёт дата рождения,
> которой на сайтах обычно нет.

## Мониторинг источников (нужно ли обновлять перечень)

`watch_sources.py` ежедневно проверяет источники по их **признакам
актуальности** и говорит, изменились ли данные с момента последней сборки
перечня — чтобы не пересобирать список вслепую.

```bash
python watch_sources.py --config config.yaml      # проверить и сообщить
python watch_sources.py --update-baseline         # принять текущее за эталон
                                                   # (после пересборки перечня)
```

Признак (флаг) подбирается под каждый источник одним запросом:

| Источник | Флаг | Метод |
|---|---|---|
| ФСБ | HTTP `ETag` / `Last-Modified` | `http_headers` |
| Минюст — иностранные агенты | `lastModified` реестра + `size` (API) | `api_registry` |
| Минюст — экстремисты | число записей + хеш содержимого | `content` |
| Росфинмониторинг | число записей + хеш содержимого | `content` |

Метод выбирается автоматически по типу источника; можно переопределить в
`config.yaml` через `options.watch_method`. Эталонное состояние хранится в
`output/source_state.json` и **не двигается само** — флаг «🔴 ОБНОВИТЬ»
держится, пока вы не пересоберёте перечень и не выполните `--update-baseline`.
Коды возврата: `0` — всё актуально, `1` — есть изменения/ошибки (удобно для
cron-алертов). Каждая проверка пишет `output/source_watch_report.json`.

## Как добавить новый источник

1. Добавьте блок в `sources:` файла `config.yaml` (`type`, `path`, `options`,
   при необходимости `mapping`).
2. Для нового *типа* источника создайте класс-наследник `BaseLoader` в
   `loaders/` и зарегистрируйте его в `_REGISTRY` ([loaders/__init__.py](loaders/__init__.py)).
